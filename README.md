# Техническое задание: десктопный VoIP-софтфон "Orbital SIP"

## 1. Цель продукта

Разработать легковесное десктопное приложение для аудио-звонков по протоколу SIP с интерфейсом в формате **Widget-first UI**.

Приложение должно:

- работать как плавающий виджет поверх других окон
- быстро раскрываться в компактный интерфейс звонка
- принимать и инициировать SIP-вызовы через WSS
- оставаться пригодным для постоянной фоновой работы

## 2. Целевые платформы и ограничения

- **Основная платформа MVP:** Windows 10/11
- **Допустимое расширение после MVP:** macOS, Linux
- **Форм-фактор:** desktop widget + compact call window
- **Основной сценарий использования:** оператор или сотрудник работает в IDE, CRM, браузере или другой бизнес-системе и принимает/совершает звонки без переключения на полноразмерный softphone

## 3. Технологический стек

- **Core:** Tauri 2.x + Rust
- **Frontend:** Vue 3 + TypeScript
- **State management:** Vue composables + reactive state
- **VoIP Engine:** SIP.js
- **Protocol:** SIP over WebSockets (WSS)
- **Media:** WebRTC
- **Styling:** Tailwind CSS
- **Animation:** Motion One или `@vueuse/motion`
- **Secure storage:** системное хранилище секретов или `tauri-plugin-stronghold`

### Зафиксированные технологические решения

- используется **Vue**, а не React/Svelte
- используется **SIP.js**, а не JsSIP
- для анимации не используется Framer Motion, так как он не является базовым выбором для Vue-стека
- основная поддерживаемая платформа MVP - **Windows / WebView2**

## 4. Основные пользовательские сценарии

### 4.1. Исходящий вызов

Пользователь раскрывает виджет, вводит номер, инициирует вызов, получает индикацию состояния вызова и элементы управления звонком.

### 4.2. Входящий вызов

При входящем вызове приложение автоматически разворачивает виджет в режим входящего звонка, отображает номер или имя абонента и показывает кнопки ответа или отклонения.

### 4.3. Работа во время активного вызова

Во время разговора пользователь должен иметь возможность:

- отключать микрофон
- ставить звонок на удержание
- переводить звонок на другого оператора
- завершать звонок
- видеть длительность и текущий статус вызова

### 4.4. Восстановление после потери сети

Если соединение временно пропадает, приложение должно пытаться восстановить SIP-регистрацию и корректно отображать состояние переподключения.

### 4.5. Смена статуса оператора

Пользователь должен иметь возможность вручную менять свой рабочий статус, чтобы быстро обозначать доступность для входящих обращений.

Поддерживаемые статусы MVP:

- `online`
- `offline`
- `lunch`
- `break`
- `training`

Для временных статусов пользователь должен иметь возможность выбрать таймер:

- 5 минут
- 10 минут
- 30 минут
- 60 минут

После завершения таймера приложение должно автоматически переводить пользователя в статус `online`, если пользователь не выбрал другой статус вручную.

## 5. Функциональные требования

### 5.1. Режимы отображения (UI States)

#### Состояние `Widget`

- круглая кнопка размером около 60x60 px
- свободное перемещение по экрану через drag-and-drop
- режим always on top
- визуальная индикация статуса регистрации:
	- зеленый ободок - зарегистрирован
	- желтый - переподключение
	- красный - ошибка регистрации или отсутствие соединения
- отображение аватара пользователя или иконки телефона
- сохранение позиции виджета между перезапусками приложения
- прилипание к левому или правому краю экрана после завершения перетаскивания

#### Состояние `Expanded`

- раскрытие по клику на виджет
- компактное окно набора номера
- цифровая панель набора номера
- поле отображения текущего номера
- список последних звонков
- возможность начать вызов из истории звонков

#### Состояние `Incoming Call`

- автоматическое раскрытие интерфейса при входящем вызове
- отображение имени или номера абонента
- кнопки `Answer` и `Decline`
- звуковой сигнал входящего звонка

#### Состояние `Active Call`

- отображение имени или номера абонента
- отображение длительности вызова
- отображение статуса вызова
- кнопки:
	- `Mute` - локальное отключение микрофона
	- `Hold` - перевод звонка в режим удержания
	- `Transfer` - перевод звонка на другого оператора
	- `End Call` - завершение звонка

### 5.2. Машина состояний вызова

Приложение должно поддерживать следующие состояния:

- `offline`
- `connecting`
- `registered`
- `incoming`
- `outgoing`
- `ringing`
- `in-call`
- `on-hold`
- `transferring`
- `reconnecting`
- `error`

Переходы между состояниями должны быть отражены как в логике SIP, так и в UI.

### 5.3. Телефония (VoIP)

- регистрация на SIP-сервере через WSS
- поддержка серверов Asterisk, FreePBX, Kamailio
- поддержка аудио-кодеков:
	- Opus - приоритетный
	- G.711 alaw
	- G.711 ulaw
- корректная работа с локальным и удаленным WebRTC-медиа-потоком
- автоматическая обработка входящих вызовов в фоне
- автоматическая попытка повторной регистрации после разрыва соединения
- обработка SIP-событий регистрации, разрегистрации, ошибки авторизации, ошибки транспорта

### 5.4. Управление звонком

#### `Mute`

- отключает локальный микрофон без завершения звонка
- UI должен показывать, что микрофон отключен

#### `Hold`

- переводит вызов в удержание
- при снятии с удержания звонок восстанавливается без пересоздания сессии

#### `Transfer`

- обязательная поддержка **blind transfer**
- целевой номер вводится пользователем вручную или выбирается из списка
- при выполнении перевода должен отображаться промежуточный статус `transferring`
- в случае ошибки перевода пользователь должен видеть сообщение об ошибке и иметь возможность вернуться к исходному вызову

Поддержка **attended transfer** может быть вынесена в post-MVP.

#### `End Call`

- завершает активный вызов по инициативе локального пользователя
- UI должен корректно возвращаться в предыдущее состояние

### 5.5. История вызовов

- хранение последних вызовов локально
- минимум 50 последних записей
- для каждой записи хранить:
	- направление вызова: `incoming`, `outgoing`, `missed`
	- номер или имя
	- дату и время
	- длительность
	- статус завершения
- возможность повторного вызова из истории

### 5.6. Статусы оператора

- пользователь может вручную менять статус из UI виджета или раскрытого окна
- обязательные статусы MVP:
	- `online`
	- `offline`
	- `lunch`
	- `break`
	- `training`
- для статусов `lunch`, `break` и `training` должен поддерживаться выбор таймера:
	- 5 минут
	- 10 минут
	- 30 минут
	- 60 минут
- активный статус должен отображаться в UI
- для временного статуса должен отображаться обратный отсчет до окончания таймера
- по завершении таймера статус автоматически меняется на `online`
- пользователь может досрочно отменить временный статус и вручную вернуть `online`
- текущий статус и оставшееся время таймера должны сохраняться при сворачивании приложения в трей
- после перезапуска приложения статус должен быть восстановлен, если срок таймера еще не истек
- если срок таймера истек во время закрытого состояния приложения, при следующем запуске статус должен быть приведен к `online`

### 5.7. Системные функции

- **Always on Top:** виджет должен быть виден поверх других приложений
- **Tray Integration:** приложение можно свернуть в системный трей без завершения SIP-сессии
- **Notifications:** нативные уведомления ОС о пропущенных вызовах и ошибках регистрации
- **Presence UI:** отображение текущего статуса оператора и таймера в виджете и в раскрытом окне
- **Auto-start:** опциональный запуск при старте системы
- **Permissions:** запрос и контроль доступа к микрофону
- **Logging:** отдельный уровень логирования для SIP/WebRTC-отладки без утечки паролей

## 6. Нефункциональные требования

### 6.1. Производительность

- потребление RAM в idle-режиме: не более 180 МБ на Windows в течение 10 минут после запуска
- запуск приложения до интерактивного состояния: не более 2 секунд на целевой машине разработки
- задержка аудио: не более 200 мс при стабильном интернете и корректной сетевой конфигурации

### 6.2. Надежность

- повторная регистрация после восстановления сети: не более 5 секунд
- входящий вызов должен быть отображен в UI не позднее 1 секунды после получения SIP INVITE
- пропущенный вызов должен сохраняться в истории не позднее 2 секунд после завершения

### 6.3. Безопасность

- пароль SIP не хранится в открытом виде в конфигурационных файлах
- учетные данные хранятся в системном защищенном хранилище
- пароли и токены не попадают в логи
- debug-логи должны маскировать чувствительные SIP-данные

## 7. Архитектурные требования

### 7.1. Разделение ответственности

- **Vue frontend:** UI, локальное состояние, анимации, обработка пользовательских действий
- **SIP layer:** регистрация, звонки, reconnect, transfer, hold, call state handling
- **Rust/Tauri layer:** управление окном, системный трей, автозапуск, уведомления, secure storage, platform-specific integration

### 7.2. Работа в фоне и проблема троттлинга

Для MVP принимается решение отключать background throttling у WebView на Windows через флаги Chromium/WebView2.

Рекомендуемые флаги:

- `--disable-background-timer-throttling`
- `--disable-renderer-backgrounding`
- `--disable-features=RendererBackgrounding,BackgroundTimerThrottling`
- `--autoplay-policy=no-user-gesture-required`

Ограничения решения:

- решение ориентировано прежде всего на Windows + WebView2
- решение снижает риск засыпания вебвью, но не отменяет системный sleep/hibernate
- при выходе системы из сна приложение должно повторно проверить регистрацию и состояние транспорта

### 7.3. Поведение после сна системы

После `sleep` / `resume` приложение должно:

- перепроверить сетевое соединение
- проверить валидность SIP-регистрации
- при необходимости выполнить повторную регистрацию
- обновить визуальный статус виджета

## 8. Этапы разработки MVP

### Этап 1: инфраструктура Tauri и Rust

- настройка прозрачного окна в `tauri.conf.json`
- реализация drag-region и свободного перемещения окна
- реализация сохранения позиции виджета
- настройка прав доступа к микрофону
- настройка always on top и tray behavior

### Этап 2: SIP-сигнализация

- интеграция SIP.js
- реализация регистрации `REGISTER`
- обработка входящих и исходящих вызовов
- обработка переподключения и ошибок транспорта

### Этап 3: Media / WebRTC

- реализация захвата микрофона
- привязка удаленного аудио-потока к скрытому элементу `<audio>`
- управление устройствами ввода и вывода
- реализация `Mute` и `Hold`

### Этап 4: UI/UX

- реализация анимации перехода `Widget -> Expanded -> Active Call`
- верстка интерфейса входящего и активного вызова
- реализация истории вызовов
- реализация UI смены статуса оператора и таймера
- реализация `Transfer`

### Этап 5: стабилизация

- тестирование reconnect-сценариев
- тестирование после блокировки экрана и resume
- тестирование уведомлений, трея и автозапуска
- профилирование потребления памяти

## 9. Критерии приемки MVP

MVP считается завершенным, если выполняются следующие условия:

- приложение запускается на Windows 10/11
- пользователь может зарегистрироваться на SIP-сервере через WSS
- пользователь может принимать и совершать аудио-звонки
- работают `Mute`, `Hold` и `Transfer` в blind transfer-сценарии
- входящий вызов корректно раскрывает интерфейс
- история вызовов сохраняется локально и доступна после перезапуска
- пользователь может менять статус между `online`, `offline`, `lunch`, `break`, `training`
- для временных статусов работают таймеры 5, 10, 30 и 60 минут с автоматическим возвратом в `online`
- приложение продолжает корректно обрабатывать регистрацию в фоне с учетом выбранных флагов WebView2
- пароль SIP не хранится в открытом виде

## 10. Риски и способы снижения

### Блокировка автопроигрывания аудио

**Снижение риска:** использовать флаг `--autoplay-policy=no-user-gesture-required` и предусмотреть первый пользовательский жест при инициализации аудио.

### Потеря связи при переключении Wi-Fi/4G

**Снижение риска:** реализовать автоматическую повторную регистрацию, обработку ошибок транспорта и обновление UI-статуса.

### NAT и отсутствие звука

**Снижение риска:** поддержка STUN/TURN в конфигурации WebRTC и обязательная диагностика ICE-состояний в логах.

### Фоновый троттлинг WebView

**Снижение риска:** использовать флаги Chromium/WebView2, а также обязательную проверку состояния SIP после resume.

### Отличия поведения на разных ОС

**Снижение риска:** ограничить MVP платформой Windows и вынести кроссплатформенную адаптацию в отдельный этап.
